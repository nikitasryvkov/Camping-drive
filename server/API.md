# Camping Drive API

API использует JSON, находится под префиксом `/api` и возвращает идентификаторы PostgreSQL `bigint` как строки. Тело успешного одиночного ответа имеет вид `{ "data": ... }`. Списки возвращают:

```json
{
  "data": [],
  "pagination": { "limit": 20, "offset": 0, "total": 0 }
}
```

Для списков доступны `limit` (1–100) и `offset`. Ошибки имеют единый формат:

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "The request contains invalid data",
    "details": { "fields": [{ "field": "slug", "message": "..." }] }
  }
}
```

## Авторизация администратора

Для `/api/pages`, `/api/news`, `/api/images` и `/api/settings` обязательна активная
сессия администратора. Без нее API возвращает `401 AUTHENTICATION_REQUIRED`. В системе
есть ровно одна роль — `administrator` с полным доступом. Публичными остаются health-маршруты,
проверка текущей сессии, вход/выход, чтение опубликованных страниц и новостей, а также
публичные настройки оформления сайта.

- `POST /api/auth/login` — вход с JSON-телом `{ "login": "...", "password": "..." }`.
- `GET /api/auth/session` — возвращает `authenticated: false` либо администратора и срок сессии.
- `POST /api/auth/logout` — удаляет текущую сессию и cookie.

При входе сервер выдает случайный непрозрачный токен в cookie с флагами `HttpOnly` и
`SameSite=Strict`; в PostgreSQL хранится только SHA-256 токена. Срок сессии задается
`ADMIN_SESSION_TTL_HOURS` (12 часов по умолчанию), пароли хранятся как scrypt-хеши с
индивидуальной солью. После пяти неудачных попыток для пары IP/логин вход блокируется на
оставшуюся часть 15-минутного окна; отдельный лимит по IP ограничивает перебор логинов и
дорогие scrypt-проверки. Резервирование попытки выполняется до асинхронной проверки пароля,
поэтому лимит действует и для параллельных запросов. Встроенный limiter рассчитан на один
процесс backend в development; в production попытки координируются через PostgreSQL,
а входной reverse proxy дополнительно ограничивает частоту по реальному IP клиента.

Локально учётная запись создается одноразовой командой `npm run admin:create`, а в Docker —
maintenance service `admin-create` с owner-подключением к БД. В обоих случаях используются
временные `ADMIN_LOGIN` и `ADMIN_PASSWORD`; минимальная длина нового пароля — 12 символов.

## Страницы и блоки

- `GET /api/public/pages/:slug` — публичная страница без авторизации. Возвращает только страницу со статусом `published` и только видимые блоки, уже отсортированные по `position`; для черновика и неизвестного URL ответ одинаковый — `404 NOT_FOUND`.
- `GET /api/public/news?limit=1..12&offset=0..1000000` — опубликованные новости и объект `pagination`; используется публичным списком и блоком последних новостей, черновики в выдачу не попадают.
- `GET /api/public/news/:slug` — отдельная опубликованная новость с текстом, обложкой и SEO-полями; для черновика, неизвестного или некорректного URL возвращается `404 NOT_FOUND`.

- `GET /api/pages` — список; фильтры `status=draft|published`, `search`.
- `POST /api/pages` — создать; обязательны `slug`, `title`.
- `GET /api/pages/:id` и `GET /api/pages/slug/:slug` — страница вместе с `blocks`.
- `PATCH /api/pages/:id` — изменить; в JSON обязателен `expectedUpdatedAt`.
- `DELETE /api/pages/:id?expectedUpdatedAt=<RFC3339>` — удалить только прочитанную версию.
- `PUT /api/pages/:id/builder` — атомарно сохранить метаданные и полный набор блоков. Обязателен `expectedUpdatedAt`; устаревшая версия получает `409 EDIT_CONFLICT`. Блоки без `id` создаются, отсутствующие в наборе удаляются.
- `GET /api/pages/:pageId/blocks` — упорядоченные блоки страницы.
- `POST /api/pages/:pageId/blocks` — создать; обязательны `type`, `content` и версия родительской страницы `expectedUpdatedAt`; `position` назначается автоматически, если отсутствует.
- `PUT /api/pages/:pageId/blocks/reorder` — атомарно изменить порядок; `blockIds` должен содержать ID всех блоков страницы без повторов, также обязателен `expectedUpdatedAt`.
- `GET /api/pages/:pageId/blocks/:blockId` — получить блок.
- `PATCH /api/pages/:pageId/blocks/:blockId` — изменить блок; в JSON обязателен `expectedUpdatedAt` родительской страницы.
- `DELETE /api/pages/:pageId/blocks/:blockId?expectedUpdatedAt=<RFC3339>` — удалить блок только из прочитанной версии страницы.

Поля страницы: `slug`, `title`, `status`, `seoTitle`, `seoDescription`, `publishedAt`. Поля блока: `type`, `position`, `isVisible`, `content` (JSON-объект). Поддерживаются типы `hero`, `text`, `image-text`, `cards`, `features`, `steps`, `stats`, `gallery`, `reviews`, `faq`, `route-map`, `cta`, `latest-news`, `marquee`; сервер проверяет структуру, числовые диапазоны и безопасные URL каждого типа. Ссылки изображений допускают только `http(s)` и root-relative URL, а кнопки дополнительно поддерживают `mailto:`, `tel:` и якоря. Поля внутри `content`, имена которых оканчиваются на `imageId` или `imageIds`, автоматически регистрируются как ссылки на таблицу изображений. Несуществующий ID отклоняется, а используемое блоком изображение нельзя удалить до удаления ссылки. JSON входа ограничен 16 КиБ для `/api/auth` и 2 МиБ для защищённых CRUD-маршрутов; проверка сессии выполняется до чтения большого защищённого тела, поэтому запрос без сессии получает `401`, а превышение лимита после входа — `413 PAYLOAD_TOO_LARGE`. Multipart `/api/images` имеет отдельный лимит 16 МиБ. Byte-budget атомарного builder payload — 2 МиБ. Для одной страницы разрешено не более 60 блоков, 300 элементов коллекций и 512 КиБ суммарного JSON-содержимого блоков; эти лимиты проверяются при сохранении, точечных изменениях и перед production-запуском опубликованных страниц.

## Новости

- `GET /api/news` — список; фильтры `status=draft|published`, `search`.
- `POST /api/news` — создать; обязательны `slug`, `title`.
- `GET /api/news/:id` и `GET /api/news/slug/:slug` — получить.
- `PATCH /api/news/:id` — изменить; обязателен `expectedUpdatedAt`.
- `DELETE /api/news/:id?expectedUpdatedAt=<RFC3339>` — удалить только прочитанную версию. Устаревшие версии получают `409 EDIT_CONFLICT`.

Поля: `slug`, `title`, `excerpt`, `content`, `coverImageId`, `status`, `publishedAt`, `seoTitle`, `seoDescription`. У опубликованной новости `excerpt` и `content` не могут быть пустыми. При первой публикации без `publishedAt` база проставляет текущее время. Явное значение `publishedAt` должно быть реальной датой в формате RFC 3339 с `Z` или смещением часового пояса; будущая дата планирует выход, и до неё материал не попадает в публичный API.

## Изображения

- `GET /api/images` — список; фильтры `mimeType`, `search`. Ответ также содержит `uploadConstraints`, чтобы UI показывал актуальные лимиты backend.
- `POST /api/images` — multipart-загрузка. Обязательное файловое поле `image`, необязательное текстовое поле `altText`.
- `GET /api/images/:id` — получить изображение.
- `PATCH /api/images/:id` — изменить `altText`; в JSON обязателен `expectedUpdatedAt`.
- `DELETE /api/images/:id?expectedUpdatedAt=<RFC3339>` — удалить прочитанную версию записи и файлы, если изображение не используется блоками, новостями или настройками.

Сервер проверяет содержимое, а не только MIME-тип клиента. Принимаются статичные JPEG, PNG и WebP; по умолчанию до 15 МБ и 40 мегапикселей (лимиты настраиваются переменными окружения). Исходник автоматически поворачивается по EXIF, конвертируется в WebP и сохраняется в размерах до 2560, 1280 и 480 px. Число одновременно принимаемых обработок задаёт `IMAGE_PROCESSING_CONCURRENCY` (по умолчанию 1, максимум 2); насыщенный backend отвечает `503 IMAGE_PROCESSING_BUSY` и `Retry-After` до приёма multipart-тела. Ответ содержит `url`, `variants`, размеры, вес и `usageCount`. Для используемого файла удаление возвращает `409 IMAGE_IN_USE`.

Перед удалением строки пути файлов транзакционно записываются в `image_deletion_queue`. API ждёт физической очистки перед ответом `204`; если volume временно недоступен, задача остаётся в очереди и повторяется фоновым worker с backoff. Тот же worker раз в час удаляет старше часа только те файлы, на которые нет записи `images`, закрывая crash-window между записью файла и INSERT.

## Настройки сайта

- `GET /api/public/settings` — публичная конфигурация шапки, контактов, меню, подвала,
  плавающих кнопок и SEO списка новостей. URL логотипа разрешается сервером через медиатеку;
  при отсутствии сохранённой настройки возвращаются безопасные встроенные значения.

- `GET /api/settings` — список; фильтры `public=true|false`, `search`.
- `GET /api/settings/:key` — получить настройку.
- `PUT /api/settings/:key` — создать или полностью заменить; обязательны поле `value` с любым JSON-значением и `expectedUpdatedAt` (`null` только при первом создании).
- `PATCH /api/settings/:key` — частично изменить; обязателен `expectedUpdatedAt`.
- `DELETE /api/settings/:key?expectedUpdatedAt=<RFC3339>` — удалить только прочитанную версию.

Ключ имеет формат `section.name`; остальные поля — `value`, `description`, `isPublic`.
Устаревшее изменение существующей настройки получает `409 EDIT_CONFLICT`; если запись уже была конкурентно удалена, сервер возвращает `404 NOT_FOUND` и не создаёт её повторно.
Для системного ключа `site` сервер проверяет полную структуру, ограничения длины и протоколы
всех ссылок. Настройку `site` нельзя сделать приватной. Выбранный `logoImageId` регистрируется
как FK-ссылка на медиатеку, поэтому активный логотип нельзя удалить до его замены.

JSON-значения блоков, вариантов и настроек проверяются рекурсивно. Числовые лексемы проверяются по сырому телу до `JSON.parse`: запрос отклоняется, если JavaScript изменил бы значение при разборе (например, слишком точную дробь, число вне диапазона или небезопасное целое).

## Служебные маршруты

- `GET /api/health` — процесс работает.
- `GET /api/health/ready` — PostgreSQL доступен, каталог загрузок доступен для записи.

Миграции из `server/migrations` применяет отдельный one-shot service `migrate`; HTTP-backend их не запускает. Backend стартует только после его успешного завершения. Номера имеют фиксированный трёхзначный формат. Изменение или удаление уже примененного файла, дублирование номера, добавление старшего по истории файла с меньшим номером и пустой каталог блокируют deployment, чтобы не допустить незаметного расхождения схемы. Мигратор использует неблокирующий advisory lock, `lock_timeout=15s` и `statement_timeout=120s`, после чего выдаёт диагностическую ошибку. Миграция `002` переносит в FK-реестр ссылки на изображения из ранее созданных JSON-блоков. Перед переносом она проверяет значения и существование изображений; при проблеме ошибка перечисляет ID блоков/изображений и подсказывает, что исправить, вместо создания скрытой несогласованности.

Интеграционный тест запускается командой `npm run test:backend:integration` при настроенных переменных PostgreSQL. В целях защиты он отказывается работать, если `PGDATABASE` не оканчивается на `_test`.
